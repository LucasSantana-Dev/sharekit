# ADR-0003: Profile composition — deferred

- **Status:** Accepted (decision = defer)
- **Date:** 2026-06-23
- **Deciders:** Lucas Santana
- **Relates to:** issue #62

## Context

`sharekit` profiles are flat and standalone: a publisher's `sharekit-profile` repo
(CLAUDE.md, skills, dotfiles) is fetched and applied as a unit. There is no way for a
profile to **compose** — to declare "also install profile B" or "extend B's setup."

Backlog issue #62 (auto-generated by a `/backlog` audit, not user-requested) proposed a
`[dependencies]` table in `sharekit.toml`. The question: build profile composition now, and
if so, in what shape?

**Demand & adoption evidence (measured 2026-06-23):**
- Repo: **1 star, 0 watchers, 0 forks, Discussions disabled** — essentially no user base yet.
- **Zero human requests** for composition/extension/reuse (only the bot-generated #62).
- The 13 features just shipped (install/update/uninstall/scan/list/rollback/inspect/…) all
  operate on a **single** profile — none implicitly require cross-profile resolution.
- The manifest parser (`smol-toml`, see ADR-0002) already supports a `[dependencies]` table,
  so adding one later is **not blocked** by tooling.

## Decision

**Defer.** Do not build profile composition now. Document the limitation in the README. If/when
it is built, prefer a **lightweight shape** — a *meta-profile* (a profile that lists other
`user@ref` profiles to install sequentially, reusing the existing apply+backup+rollback,
last-write-wins on file conflicts) or a *single `extends`* (a profile extends exactly one base)
— **not** a full `[dependencies]` DAG with multi-dependency conflict resolution and version
reconciliation.

A `decision-critic` review returned NEEDS_REVISION, endorsing deferral but flagging a **circular
revisit trigger** ("revisit when users ask" while having no way to sense demand) and warning
that "no reactions" might over-read demand on a too-new tool. Verifying adoption confirmed the
concern *and* strengthened the decision: with ~0 users there is no one to survey, so a complex
speculative feature is premature. The trigger below is rewritten to be concrete and measurable.

## Alternatives considered

- **Full `[dependencies]` DAG** — *rejected.* Largest effort (recursive fetch, dependency
  ordering, cycle detection, two-deps-touch-one-file conflict resolution, version
  reconciliation) for a speculative, zero-demand feature on a ~0-adoption tool.
- **Single `extends` (one base profile)** — *not now;* the preferred shape **if** built later.
- **Meta-profile (client-side sequential install of a listed set)** — *not now;* the thinnest
  shape, arguably needs no real composition engine. Co-preferred for "if built later."
- **Build it now** — *rejected.* Violates the project's "don't make demand-blind investments"
  rule; 1 star + 0 requests makes this textbook premature complexity.

## Consequences

- **Positive:** no premature complexity; the `sharekit.toml` schema stays open (smol-toml
  handles a future `[dependencies]`/`extends` key without migration); engineering stays focused
  on the features users of a young tool actually hit.
- **Negative / known limitation:** a publisher can't reuse or extend another profile's base; the
  workaround is to `install` multiple profiles in sequence (or copy files into one's own
  profile). Sequential installs are **last-write-wins** on overlapping files — recoverable via
  `sharekit rollback`, but a silent precedence surprise. This is documented in the README as a
  limitation so users aren't caught off guard.
- **Neutral:** the "if-built" shape is pre-picked (meta-profile / single-extends), so the
  downstream decision is de-risked, not merely postponed.

## Revisit when (concrete trigger)

Re-open this decision if **any** holds:
- A **human** files an issue or Discussion requesting profile reuse / extension / combination
  (any phrasing: "extend a base", "share a skill library", "combine profiles").
- Adoption reaches a real threshold (**≥25 stars or ≥3 distinct published `sharekit-profile`
  repos in the wild**) **and** a concrete shared-base use-case is articulated.
- A future feature begins to **implicitly** need cross-profile resolution (e.g. `update` having
  to reconcile versions across profiles).

At revisit: pick **meta-profile** or **single `extends`** and run a 1-hour prototype before
committing; do **not** default to a full DAG.
